## 🍎 Apple Silicon & Local AI

### Ollama 0.19 Adds MLX Support for Apple Silicon

Ollama released preview support for Apple's **MLX framework** (v0.19), delivering major performance gains on M-series Macs:

- **1.6x faster prefill speed** (prompt processing)
- **Nearly 2x faster decode speed** (response generation)
- **M5-series chips see largest improvements** thanks to new GPU Neural Accelerators
- **Preview limitations:** Currently supports only Qwen3.5-35B-A3B; requires 32GB+ RAM
- Ollama also added smarter memory management and NVIDIA NVFP4 format support

**Impact:** OpenClaw, Claude Code, OpenCode, Codex users should see "noticeably more responsive" performance during extended use. Combined with MLX's zero-copy operations and unified memory allocation, Apple Silicon Macs increasingly competitive with cloud APIs for local inference.

**Sources:** [Ars Technica](https://arstechnica.com/apple/2026/03/running-local-models-on-macs-gets-faster-with-ollamas-mlx-support/), [MacRumors](https://www.macrumors.com/2026/03/31/ollama-now-runs-faster-apple-silicon-macs/), [Ollama Blog](https://ollama.com/blog/mlx)

### Apple M5 Pro/Max Launch: Chiplet Architecture + Neural Accelerators

Apple announced **M5 Pro and M5 Max** — the company's first **chiplet-based consumer silicon** (Fusion Architecture):

- **Shared 18-core CPU** (6 "super cores" + 12 performance cores) across Pro and Max
- **M5 Pro:** 20-core GPU | **M5 Max:** 40-core GPU
- **Neural Accelerator in each GPU core** → 4x faster LLM prompt processing vs. previous gen
- **M5 Ultra inevitable:** Two M5 Max dies fused via UltraFusion → 80-core GPU, up to 256GB unified memory, potentially 1.2TB/s bandwidth

**Implications:**
- Same CPU die scales from MacBook Pro → Mac Studio → Mac Pro → Private Cloud Compute servers
- 400B+ parameter model at 4-bit (or 120B+ at FP16) fitting entirely in single machine memory
- Apple demonstrated 19-27% faster token generation M5 vs. M4

**Sources:** [Apple Newsroom](https://www.apple.com/newsroom/2026/03/apple-debuts-m5-pro-and-m5-max-to-supercharge-the-most-demanding-pro-workflows/), [BEP Research](https://bepresearch.substack.com/p/the-other-memory-wall-part-2)

### Qwen3.5-397B Running Locally on M3 Ultra via MLX

Developer **Ivan Fioravanti** (@ivanfioravanti) posted benchmarks of **Qwen3.5-397B-A17B** (397B parameters MoE, 17B active) running via MLX on M3 Ultra (512GB):

- **38 tokens/second** at short context lengths
- **24 tok/s** at 128K context
- At 4-bit quantization; faster than most cloud API endpoints
- On M5 Ultra, extrapolated to 45-50 tok/s at short context (past GPT-4o API response times)

**Developer Adrien Grondin** posted **Qwen 3.5 2B** (6-bit quantization) running **on-device on iPhone 17 Pro** via MLX. Post went viral: 1.8M views; Alibaba amplified.

**Analysis (BEP Research):** These two data points bracket the entire inference spectrum — 2B model on phone for instant/private responses, 397B model on workstation for frontier-class reasoning. Both on MLX, both leveraging unified memory. "MLX on Apple Silicon isn't just 'fast.' It's enabling workloads that the x86 + discrete GPU ecosystem physically cannot serve."

**Source:** [BEP Research](https://bepresearch.substack.com/p/the-other-memory-wall-part-2)

### Apple's Three-Layer AI Infrastructure

**BEP Research analysis:** Apple upgraded Private Cloud Compute (PCC) to M5 chips, skipping M3 and M4 entirely. Combined with M5 chiplet architecture and MLX community momentum, Apple now has a three-layer inference architecture running **same silicon DNA from phone to server rack**:

1. **On-Device:** Small models (2B-7B) on iPhone/iPad/Mac via MLX and Foundation Models
2. **Private Cloud Compute:** Medium-to-large models (30B-200B+) on M5-powered PCC servers (manufactured in Houston, Texas)
3. **Partner (Gemini):** Frontier models for tasks requiring maximum capability

**Key insight:** "No other company has this. Google's cloud runs on TPUs while your Pixel runs on Tensor chips — different architectures, different software stacks. Apple has one architecture, one framework, one memory model, from phone to cloud. That's co-design at infrastructure scale."

**Source:** [BEP Research](https://bepresearch.substack.com/p/the-other-memory-wall-part-2)
